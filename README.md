# NovelForge

NovelForge 是一个面向长篇小说创作的 LLM 写作工作台，核心围绕持久化项目存储、检索增强、结构化工作流，以及未来的多 Agent 协作能力构建。

它更适合长期创作场景，例如同人文、网文、长篇原创小说，而不是一次性聊天生成。项目重点关注设定持续维护、上下文记忆、分层规划和迭代修订。

## 项目定位

大多数 LLM 写作工具更偏向短对话和单次生成。
NovelForge 想解决的是另一类问题：

- 基于项目的长期写作
- 长期记忆与设定维护
- 从总纲到章节的完整工作流
- 检索增强上下文注入
- 结构化审阅与校验
- 为后续工作流运行时和多 Agent 协作预留架构

这个项目同时也是：

- 一个实际可用的小说写作工具
- 一个用于学习 LLM 应用、RAG、工作流与 Agent 系统的实验平台

## 当前状态

NovelForge 已经不再是早期原型。

当前成熟度大致可以概括为：

- V1 写作工作台：已实现

- 项目感知的页面头部与统一卡片化界面样式

## 设计原则

NovelForge 遵循几条核心原则：

1. 持久化优先于智能化
2. 工作流优先于多 Agent
3. 技能化优先于自治化
4. 一切以项目为中心
5. 通过 OpenAI-compatible 接口保持模型独立性

## 架构概览

当前高层流程：

```text
用户
-> Streamlit UI (app.py)
-> Skill Layer (skills.py)
-> Prompt Layer (prompts.py)
-> LLM Interface (llm.py)
-> OpenAI-compatible API
-> Memory / Storage / Retrieval
```

主要文件职责：

- `app.py`：界面与交互流程
- `skills.py`：写作与分析能力
- `prompts.py`：提示词模板与拼装
- `llm.py`：模型抽象与 API 集成
- `memory.py`：持久化存储与项目数据管理
- `schemas.py`：结构化输出契约与校验
- `retrieval.py`：索引、检索与上下文格式化

## 典型工作流

NovelForge 支持直接生成，也支持“先讨论，再生成”的规划方式。

典型章节流程：

1. 讨论故事或章节方向
2. 生成总纲或章节细纲
3. 生成章节正文
4. 审阅章节质量与一致性
5. 更新故事记忆
6. 需要时查看分析报告或检索证据
7. 需要时执行章节评估，或从失败流水线运行继续恢复

系统也支持一键流水线：

```text
Plan -> Write -> Review -> Update Memory
```

## 资料、设定与检索

NovelForge 内置项目级检索层，可同时处理内部写作资料和外部参考资料。

当前界面把三类入口拆开：

- `项目资源`：文件级管理入口，用来查看、编辑、删除项目中的大纲、正文、报告、运行快照和资料文件。
- `核心设定`：短而稳定的故事状态，生成时会优先注入，例如主设定、角色关系、时间线和硬性约束。
- `资料录入` / `检索中心`：前者负责导入和结构化提取，后者负责索引、检索测试、调试和冲突裁决。
- `长篇资料批次管理`：整本 txt 上传后，会记录每个片段是否已导入、已提取、提取失败或仍待处理。
- `资料指纹`：记录文本内容哈希、文件名、总字数和片段数，用来识别重复上传的整本资料。
- `待确认结构化知识`：资料提取结果可以先暂存，确认后再进入正式结构化知识并参与检索。
- `结构化知识整理`：长篇资料批量提取后，用来合并同名角色、技能、地点和其它重复条目。
- `资料包报告`：把已确认结构化知识整理成可保存、可检索的项目资料总览。

当前工作台导航也已经按使用场景分组：

- `工作台`：项目总览、模型配置、生成规则、项目资源
- `规划`：创作配置，以及按当前故事配置动态展开的大纲、分卷、剧情段和细纲入口
- `写作`：动态生成、一键流水线、写章节、章节审阅、章节评估、一致性分析
- `资料`：核心设定、资料录入、检索中心

`项目总览` 已经从单纯统计页升级为工作台首页，除了显示项目指标，也提供继续生成、章节流水线、整理资料和查看资源等快捷入口。

当前支持的检索能力包括：

- 项目知识检索
- 原作 / 参考资料检索
- 长篇文本按章节标题或字数切分后批量导入
- 长篇资料批次进度跟踪，支持继续导入、继续提取和失败重试
- 重复上传检测，支持绑定到已有批次继续处理
- 文档切块索引
- 语义向量检索
- 词法 + 语义混合排序
- 来源权威度加权
- 按作用域分组证据
- 当项目证据与外部证据冲突时给出提示
- 保存冲突裁决，后续检索会把裁决作为项目知识召回
- 可选检索调试信息，帮助排查召回和排序原因
- 从粘贴资料中提取结构化知识，确认后保存到项目结构化知识
- 将提取结果加入待确认队列，支持批量确认、丢弃和高级编辑
- 按分类整理结构化知识，支持重复检测、合并预览、删除和原始数据编辑
- 从长篇资料片段中分批提取结构化知识，避免一次性处理过长文本
- 从已确认知识生成资料包报告，保存后进入检索索引
- 结构化知识会进入检索索引，供大纲、章节、审阅、分析和评估流程召回
- 已批准的创作配置讨论工件也会进入检索索引，供动态结构、章节规划和写作流程参考

结构化知识第一版支持：

- 角色知识
- 物品与道具
- 技能与能力
- 世界观规则
- 地点与组织
- 事件与时间线
- 角色关系
- 写作风格、对白风格和写作手法
- 硬性约束

## 故事空间

NovelForge 支持在同一项目内创建多个独立故事。项目级资源（基础设定、结构化知识库、全局规则、项目规则、外部资料与检索索引）在所有故事间共享，而每个故事可以拥有：

- 独立的创作配置（任务性质、参考强度等）
- 独立的记忆覆盖（只影响当前故事的设定差异）
- 独立的大纲、分卷、剧情段、章节、审阅、评估和流水线记录
- 独立的章节摘要积累

这样你可以导入一次原作资料，然后用同一个角色和世界观写出多个不同走向的故事——原作续写、平行世界、角色穿越、短篇番外等，互不干扰。

侧边栏会保存当前激活故事；项目总览、资源浏览器、动态生成、规划、写作、审阅、分析、评估和流水线都会跟随当前故事读写。首次使用已有项目时，系统会自动将现有内容迁移为默认故事，不会丢失任何数据。

## 创作配置

每个故事可以保存一份创作配置，用于描述当前故事更适合走哪种生成路径。

当前支持配置：

- 任务性质：主线故事、番外、续写、前传、穿越、补完、片段，也可自定义
- 目标篇幅与目标字数，均可自定义
- 生成层级：只生成正文、短篇结构、章节计划、分卷/剧情段、完整长篇流程
- 资料参考强度：轻参考、中参考、强参考、严格原作、主要参考文风
- 重点参考方向：角色、世界观、剧情事件、道具能力、时间线、写作风格、对白风格、写作手法、硬性约束，也可补充自定义标签
- 资料冲突处理策略，可自定义

`创作任务向导` 可以用更直接的中文问题生成配置：这次想写什么、大概篇幅、希望系统产出什么、参考资料强度、冲突处理策略和补充说明。

如果一开始还不确定该如何选择，也可以先使用 `创作配置` 页内的讨论辅助，用自然语言描述目标，再把讨论得出的推荐配置回填到表单中继续微调，或直接批准为当前故事创作配置并保存对应讨论工件。

这份配置会注入主要生成、讨论、审阅和分析提示中，用于影响模型对长度、结构层级和资料参考比例的判断。完整长篇的自动编排仍在后续规划中。

新故事在未保存创作配置前，规划区只显示 `创作配置`。保存配置后，规划入口会按目标篇幅和生成层级展开：长篇显示全书 / 分卷 / 剧情段 / 细纲，短篇或片段只显示轻量规划入口，避免出现误导性的长篇工具。

应用内的 `动态生成` 页面已经提供第一版可执行入口：

- 只生成正文
- 短篇结构 + 正文
- 章节计划 + 正文
- 根据短篇、番外、续写、前传、穿越、补完、片段等任务性质自动走轻量流程

完整长篇仍建议使用全书大纲、分卷、剧情段、章节细纲和一键流水线分步推进。

## 项目存储结构

每个项目存放在 `data/projects/` 下。项目级资源（基础记忆、知识、规则、外部资料、检索索引）为所有故事共享，故事级内容独立存储在 `stories/{story_id}/` 下。

```text
data/
  global_rules.json
  projects/
    your_project/
      memory.json                  ← 共享基础设定
      rules.json                   ← 共享项目规则
      knowledge/                   ← 共享结构化知识库
      retrieval/                   ← 共享检索索引与向量
      stories/
        index.json                 ← 故事列表与激活状态
        default/                   ← 默认故事（存量迁移后）
          creative_profile.json
          creative_profile.discussion.json
          memory_overrides.json    ← 仅本故事的设定差异
          chapter_summaries.json
          outline.md
          outline.discussion.json
          volumes/
          arcs/
          chapter_outlines/
          chapters/
          reviews/
          analysis/
          evaluation/
          runs/
          retrieval/               ← 故事级冲突裁决
```

这样可以把规划、正文、审阅、分析和检索工件统一挂在同一个项目下，而不是散落在聊天记录中。

新增工件包括：

- `stories/`：故事空间目录，每个故事独立存储
- `stories/index.json`：故事列表索引
- `memory_overrides.json`：故事级记忆差异（仅保存与本故事不同的设定）
- `chapter_summaries.json`：故事级章节摘要
- `arcs/arc_xxx.chapter_plan.json`：剧情段章节分配计划
- `creative_profile.json`：故事级创作配置
- `creative_profile.discussion.json`：已批准的创作配置讨论工件，包含推荐配置与讨论结论
- `knowledge/*.json`：确认后的结构化知识记录
- `knowledge/pending.json`：待确认结构化知识队列
- `analysis/source_package.md`：基于结构化知识生成的资料包报告
- `evaluation/chapter_xxx.md` / `.json`：章节评估报告和结构化评分
- `retrieval/conflict_resolutions.json`：检索冲突裁决记录
- `runs/*.json`：可恢复流水线运行快照

## 安装与运行

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 配置模型访问

你可以通过两种方式配置模型：

- 手动编辑 `.env`
- 在应用内使用 `模型配置` 页面创建并切换多套配置档案

典型环境变量：

```env
LLM_API_KEY=
DEEPSEEK_API_KEY=
LLM_BASE_URL=https://api.deepseek.com
LLM_MODEL=deepseek-v4-flash
LLM_EMBEDDING_MODEL=text-embedding-3-small
```

### 3. 启动应用

```bash
streamlit run app.py
```

## 本地 Windows 便携版构建

NovelForge 也可以打包成一个本地 Windows 便携版，双击后自动启动 Streamlit 服务并打开浏览器。

### 发布形态

目标分发形式为：

- `NovelForge.exe` 作为小型启动器
- 随包附带 `.venv` 运行时
- 保留项目源码文件
- 使用本地 `data/` 目录存放项目数据

用户启动 `NovelForge.exe` 后，程序会：

1. 在 `127.0.0.1` 启动本地 Streamlit 服务，优先尝试 `8501`
2. 等待应用可访问
3. 自动打开浏览器

### 构建步骤

1. 创建并准备本地虚拟环境：

```bash
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt
```

2. 在 PowerShell 中运行构建脚本：

```powershell
.\build_release.ps1 -Version v0.1.0
```

3. 脚本会自动：

- 在 `.venv` 中安装 `pyinstaller`
- 使用仓库内的 `NovelForge.spec` 作为 PyInstaller 构建配置
- 根据 `launcher.py` 构建 `NovelForge.exe`
- 组装 `release/NovelForge-Portable/`
- 生成 `release/NovelForge-windows-portable-v0.1.0.zip`
- 在 `release/` 下保存本地构建日志

### 使用说明

- 建议把便携版解压到可写目录，例如 `D:\Apps\NovelForge\`
- 不建议放在 `C:\Program Files\` 这类受保护目录
- 用户数据会保存在同目录下的 `data/` 和 `.env`
- 启动器会优先使用 `8501`，必要时会自动回退到附近端口
- 如果启动失败，可查看应用目录中的 `launcher.log`
- 如果候选端口之一被其他应用占用，启动器会自动尝试下一个端口，而不是误打开错误页面
- 构建日志会保存在 `release/build_release-<version>.log`

## 开发说明

项目在职责划分上保持明确边界：

- UI 逻辑尽量保持在 `app.py` 的轻量层面
- 新写作能力优先通过 `skills.py` 扩展
- Prompt 工程集中在 `prompts.py`
- 持久化逻辑集中在 `memory.py`
- 模型适配尽量只改 `llm.py`
- 结构化 LLM 输出优先通过 `schemas.py` 定义

更详细的实现说明和路线规划请查看 `project.md`。
